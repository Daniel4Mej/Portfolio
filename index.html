Problems I Had to Solve
E-stop condition in Teensy firmware

The E-stop built into the Teensy firmware wasn't designed for a different calibration process. The fix was to ignore the interrupt response entirely, letting the code continue even after the E-stop button was pressed without having to manually restart the code.

Movement command was overwriting good position data

The movement command was sending empty zero positions right after restarting the program, removing the Teensy's built in step counters. The fix was to have the movement command send a position as a new command rather than trying to overwrite the old position coordinates.

Unequal joint limits broke the step counter math

For example J2 would move to 42°, and after the program resets the value would internally return back to 90°, so anything that reloaded the original value subsequently made moves go the wrong direction. Fixed by resending position coordinates before every move so that the coordinates stay consistent.

Imaginary collision faults stopping calibration early

Position coordinates would change axis limits mid-session, causing the encoder to drift which the Teensy would flag as a collision. Fixed by running all joints in open loop during calibration, stopping the Teensy from detecting an imaginary collision.

Physical axis limit causing rejections on J4 and J6

If the position degree started from +84°, giving the Teensy a command to move to -170° would cause 254° of travel, resulting in the Teensy rejecting it. Fixed by returning to home between the two direction tests so each movement command starts from 0°.

Stale error strings polluting position queries

Leftover Teensy errors in the serial buffer would cause any further positions to be compromised. Fixed by filtering and discarding any response from the Teensy that wasn't a valid position, reading and re-reading until a valid position query came through.

AI-Powered Facial Recognition Door Handle

A smart lock I developed using Raspberry Pi, machine learning libraries, and my own face encodings. It locks and unlocks using servo motor functions that I integrated, and I also added a 2-second confirmation window for added security.

Computer Vision Embedded Systems Raspberry Pi Python Servo Motor SolidWorks 3D Printing OpenCV
Why I Built It

Commercial smart locks are expensive, require Wi-Fi, and lock you into an app ecosystem. We wanted to build something that worked offline, cost under $150, and didn't need a phone connection to use. The four goals we had were using facial recognition as the main unlock, having a keypad as a backup, no Wi-Fi dependency, and a portable door mounted design.

How It Works
Facial recognition

Live frames come in at 640×480, faces get encoded and compared against stored encodings. Encodings are preloaded at startup so the live comparison is just a callback to the saved encodings. New users can be added at any time without touching any code. The live detection had roughly 150 to 300ms detection latency on the Raspberry Pi hardware with stable recognition under controlled lighting.

2-second confirmation timer

The lock would only open if the same face is detected for 2 full seconds. This stops photo fraud, accidental unlocks, and false positives. If the recognition stops mid detection the timer resets.

Keypad backup

This was added as the result of real world cases where facial recognition would struggle, like low light, bad angles, or facial obstructions. The code can be changed by the user after inputting the master code and there is a lockout feature after repeated failed attempts.

Servo actuation

The servo is actuated by the Raspberry Pi GPIO 18. On unlock it rotates 90°, holds for 2 seconds, then returns to the locked position automatically.

Challenges I Had to Work Through
Lighting messing with recognition

Poor lighting caused the detection to miss faces or return bad facial matches. Fixed by capturing encodings under multiple lighting conditions during setup and using the 2-second timer to filter out unstable detections.

Camera hanging on startup

If the camera object wasn't properly released from a previous session the Pi would just sit there doing nothing. Fixed by adding retry logic with up to 3 attempts and a cleanup function between each session.

Servo jitter

The default GPIO pulse range didn't match the servo spec, causing constant jitter and overcurrent issues. Manually setting the pulse widths to match the servo spec fixed the jitter.

Detection lag on the Raspberry Pi

Running at high resolution was too slow for live detection. Dropping to 640×480 and preloading encodings reduced latency to real time performance suitable for a door lock.

Deutsch-Jozsa Algorithm on Classical Machines

A paper I co-wrote that examines whether the Deutsch-Jozsa algorithm proves quantum superiority or if a classical computer can replicate the same result. By using Qiskit software, we discovered that a classical computer can simulate the same results as a quantum computer, only because it uses a transparent oracle.

Read Research ↓
Quantum Computing Research Qiskit Python Algorithm Analysis
What We Did

We implemented a classical simulation of the Deutsch-Jozsa algorithm in Qiskit to evaluate whether quantum advantage is true under different oracle assumptions. The goal was to evaluate the claim rather than assume its validity. We ran experiments for up to 20 qubits, analyzed circuit structure differences between quantum and classical methods, and compared outputs across both constant and balanced oracles.

My Contributions
Built the classical simulation in Qiskit using Python
Designed oracle configurations for both constant and balanced function types
Ran and verified experiment with outputs up to 20 qubits

What the Algorithm Does

Deutsch-Jozsa determines whether a function is constant (always returns 0 or 1) or balanced (returns 0 half the time and 1 the other half). A quantum computer solves this in a single query regardless of the input size. Using classical methods you would need to check half the inputs plus one before you could be certain. The algorithm is considered a method to show quantum computing superiority. That's the claim we tested.

What We Found
Classical simulation works for n less than 20

Both constant and balanced oracle types returned the correct answer every time in a single query, matching the quantum equivalent exactly. Simulation runtime did increase exponentially with increased qubit counts (O(2ⁿ)), which limited practical testing to n less than 20 on available hardware.

In our simulation the oracle is not actually a true black box

To simulate it classically you have to be able to see the CNOT gates inside the oracle. In a quantum computer the oracle is treated as opaque, meaning you can't see inside of it. That's a fundamentally different assumption, so the comparison isn't really a fair comparison. The classical simulation only works because it gets information that the quantum version wouldn't have.

The claimed quantum speedup depends entirely on the oracle model

Johansson (2017) and Yavuz and Yadav (2023) both reached the same conclusion independently. They found that quantum speedup is not universally applicable. It depends entirely on the oracle assumption. If you were to change that assumption then the argument does not hold under the modified oracle assumptions.

The Key Tradeoff

The question isn't necessarily about quantum vs. classical in general. It's more about what computing model you're working with. Transparent oracle means classical computing outperforms quantum. Truly opaque oracle means quantum computing has a provable advantage. Most real problems would fall somewhere in between, which is why this distinction matters when evaluating where quantum hardware is actually worth using.

Qiskit Implementation
n = 10  # number of query qubits

# Choose oracle type at random — constant or balanced
oracleType, oracleValue = np.random.randint(2), np.random.randint(2)
if oracleType == 0:
    print("Oracle: constant, returns", oracleValue)
else:
    a = np.random.randint(1, 2**n)  # hidden parameter for balanced oracle
    print("Oracle: balanced")

# Build the circuit
qr = QuantumRegister(n + 1)
cr = ClassicalRegister(n, name='res')
djCircuit = QuantumCircuit(qr, cr)

# Superposition on query register
for i in range(n):
    djCircuit.h(qr[i])

# Flip and Hadamard on ancilla
djCircuit.x(qr[n])
djCircuit.h(qr[n])
djCircuit.barrier()

# Oracle — constant: identity or X gate / balanced: CNOT against hidden bitstring
if oracleType == 0:
    if oracleValue == 1:
        djCircuit.x(qr[n])
    else:
        djCircuit.id(qr[n])
else:
    for i in range(n):
        if (a & (1 << i)):
            djCircuit.cx(qr[i], qr[n])  # CNOT gates visible — transparent oracle

djCircuit.barrier()

# Second Hadamard + measure — all zeros = constant, any nonzero = balanced
for i in range(n):
    djCircuit.h(qr[i])
djCircuit.measure(range(n), range(n))

# Run with StatevectorSampler
sampler = StatevectorSampler()
results = sampler.run([djCircuit]).result()
plot_histogram(results[0].data.res.get_counts())

Takeaway

This work reinforced the importance of clearly defining system assumptions when evaluating algorithmic performance. It's a principle that applies directly to engineering system design.
